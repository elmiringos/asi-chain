# k6 Load Testing — ASI-Chain

Custom k6 binary built with the `xk6-asi-chain` Go extension. All deploy signing happens in Go (secp256k1 + blake2b-256 + protobuf encoding) — no JS crypto needed.

---

## Architecture

```
k6/Dockerfile (multi-stage)
├── Stage 1: golang:1.25-alpine
│   ├── installs xk6
│   ├── copies k6-ext/ (Go extension)
│   └── builds custom k6 binary with xk6 build v0.55.0 --with xk6-asi-chain=.
└── Stage 2: alpine:3.19
    ├── copies /usr/bin/k6 from builder
    └── copies k6/src/ → /scripts/
```

The resulting image is `k6-tests:latest`, run as a Kubernetes Job in the `monitoring` namespace.

## xk6-asi-chain Extension (`k6-ext/`)

Go module `xk6-asi-chain` exposes the `k6/x/asichain` JS module.

### Files

| File | Contents |
|------|----------|
| `module.go` | Module registration + `Exports()` wiring |
| `sign.go` | `SignDeploy` — blake2b-256 + secp256k1 DER signature |
| `terms.go` | `HelloWorldTerm`, `TransferTerm` — Rholang term builders |
| `blocks.go` | `WaitForBlock`, `ParseDeployId` — block confirmation |

### API

```javascript
import { signDeploy, helloWorldTerm, transferTerm, waitForBlock, parseDeployId } from "k6/x/asichain";
```

| Function | Signature | Description |
|----------|-----------|-------------|
| `signDeploy` | `(deployData: object, privateKeyHex: string) → {sig, sigAlgorithm, deployer}` | Signs deploy fields with secp256k1; returns hex sig + compressed public key |
| `helloWorldTerm` | `() → string` | Returns `new stdout(rho:io:stdout) in { stdout!("Hello, Blockchain!") }` |
| `transferTerm` | `(fromAddr, toAddr: string, amount: int) → string` | Returns ASI vault transfer Rholang term |
| `waitForBlock` | `(nodeUrl: string, afterBlockNumber: int, timeoutSec: int) → int` | Polls `/api/blocks/1` every 500ms until block number increases; returns new block number or -1 on timeout |
| `parseDeployId` | `(body: string) → string` | Parses JSON response body to extract deploy ID |

**Signing details**: `deployData` fields signed (in protobuf order): `term`, `phloLimit`, `phloPrice`, `validAfterBlockNumber`, `timestamp`, `shardId`. Hash: blake2b-256. Algorithm: secp256k1 DER.

## Script Library (`k6/src/lib/`)

### `deploy.js`

```javascript
import { sendDeploy, deployAndCheck, getValidAfterBlockNumber, HELLO_WORLD_TERM, transferTerm } from "./lib/deploy.js";
```

| Export | Description |
|--------|-------------|
| `HELLO_WORLD_TERM` | Pre-built HelloWorld Rholang term (module-level constant) |
| `transferTerm(from, to, amount)` | Build transfer term |
| `getValidAfterBlockNumber(nodeUrl)` | `GET /api/blocks/1` → current block number; call in `setup()` |
| `sendDeploy(nodeUrl, term, validAfterBlock, privateKey, shardId)` | Signs + `POST /api/deploy`; returns k6 response |
| `deployAndCheck(nodeUrl, term, validAfterBlock, privateKey, shardId, label)` | `sendDeploy` + `check()` for status 200 and non-empty body |

### `summary.js`

```javascript
import { annotateTestRun } from "./lib/summary.js";
```

`annotateTestRun(data, scriptName, [grafanaUrl])` — posts a region annotation to Grafana covering the test run window (start = `Date.now() - testRunDuration`, end = `Date.now()`). Uses Basic auth `admin:admin`. Annotations are stored in Grafana SQLite and **persist independently of Prometheus metric retention**.

Call from `handleSummary`:

```javascript
export function handleSummary(data) {
  annotateTestRun(data, "my-script");
  return {};
}
```

### `management.js`

Utilities for interacting with the Management API (not used by default k6 scenarios).

## Scripts

All scripts live in `k6/src/`. The `testid` tag is injected at job creation time via `--tag testid=<job-name>` and used for per-run filtering in Grafana.

### `hello-world.js`

Simple HelloWorld deploy loop.

```
VUS=5, DURATION=30s (defaults)
Thresholds: http_req_failed < 5%, p95 < 10s
```

### `transfer.js`

Token transfer deploy loop. Requires `FROM_ADDR` and `TO_ADDR`.

```
VUS=5, DURATION=30s
Thresholds: http_req_failed < 5%, p95 < 8s
Env vars: FROM_ADDR, TO_ADDR, AMOUNT (default: 1)
```

### `smoke.js`

Single-VU sanity check: one HelloWorld + one transfer (if addresses set).

```
VUS=1, DURATION=60s
Thresholds: http_req_failed < 10%, p95 < 15s
```

### `scenarios/hello-world-repeat.js`

High-throughput HelloWorld: N deploys per VU iteration without sleep.

```
VUS=3, DURATION=60s, REPEAT=5 (deploys per iteration)
Thresholds: http_req_failed < 5%, p95 < 10s
```

### `scenarios/confirmed-deploy.js`

Sends deploys and waits for block confirmation. Measures true end-to-end confirmation latency.

```
VUS=2, DURATION=120s, CONFIRM_TIMEOUT=30s
Thresholds: http_req_failed < 5%, p95 < 10s, deploy_confirmation_ms p95 < 30s, deploy_unconfirmed_total < 5
```

**Custom metrics:**

| Metric | Type | Description |
|--------|------|-------------|
| `deploy_confirmation_ms` | Trend | Time from HTTP 200 to new block |
| `deploy_confirmed_total` | Counter | Deploys confirmed in a block |
| `deploy_unconfirmed_total` | Counter | Deploys that timed out |

## Running Tests

### Build the image

```bash
sudo make k6-build
```

Must rebuild after any change to `k6/src/` or `k6-ext/`.

### Run a script

```bash
# General syntax
sudo make k6-run K6_SCRIPT=<script> [K6_VUS=N] [K6_DURATION=Xs] [K6_NODE=validatorN]

# Examples
sudo make k6-run K6_SCRIPT=hello-world K6_VUS=5 K6_DURATION=60s
sudo make k6-run K6_SCRIPT=transfer K6_VUS=3 K6_DURATION=30s
sudo make k6-run K6_SCRIPT=scenarios/hello-world-repeat K6_VUS=3 K6_DURATION=60s
sudo make k6-run K6_SCRIPT=scenarios/confirmed-deploy K6_VUS=2 K6_DURATION=120s

# Shortcuts
sudo make k6-smoke    # smoke, 1 VU, 60s
sudo make k6-stress   # hello-world, 20 VUs, 5m
sudo make k6-confirm  # scenarios/confirmed-deploy, 2 VUs, 120s
```

### View logs

```bash
make k6-logs
```

### Clean up finished jobs

```bash
make k6-clean
```

## Kubernetes Job (`k6/job.yaml`)

The job template uses placeholders substituted by `make k6-run`:

| Placeholder | Substituted with |
|-------------|-----------------|
| `REPLACE_JOB_NAME` | `k6-<script-slug>-<YYYY-MM-DD-HH-MM-SS>` (e.g. `k6-hello-world-2026-03-30-11-10-05`) |
| `REPLACE_SCRIPT` | `K6_SCRIPT` value (e.g. `hello-world`) |
| `REPLACE_NODE_URL` | `http://<K6_NODE>-0.<K6_NODE>.asi-chain.svc.cluster.local:40403` |
| `REPLACE_PRIVATE_KEY` | `VALIDATOR1_PRIVATE_KEY` from `.env` |
| `REPLACE_FROM_ADDR` | `VALIDATOR1_PUBLIC_KEY` |
| `REPLACE_TO_ADDR` | `VALIDATOR2_PUBLIC_KEY` |
| `REPLACE_VUS` | `K6_VUS` |
| `REPLACE_DURATION` | `K6_DURATION` |

**Key env vars set on the job:**

```yaml
K6_PROMETHEUS_RW_SERVER_URL: http://prometheus.monitoring.svc.cluster.local:9090/api/v1/write
K6_PROMETHEUS_RW_TREND_STATS: p(50),p(95),p(99),count,sum
```

k6 args include `--out experimental-prometheus-rw` and `--tag testid=<job-name>`.

Jobs are auto-deleted 5 minutes after completion (`ttlSecondsAfterFinished: 300`).

## Prometheus Metrics

All k6 metrics are pushed via remote write with label `testid=<job-name>`.

Standard k6 metrics available in Prometheus:

| Metric | Description |
|--------|-------------|
| `k6_http_reqs_total` | Total HTTP requests |
| `k6_http_req_failed_total` | Failed requests (non-2xx) |
| `k6_http_req_duration_p50` / `_p95` / `_p99` | Latency percentiles |
| `k6_vus` | Active virtual users |
| `k6_iterations_total` | Completed iterations |

Custom metrics (from `confirmed-deploy` scenario):

| Metric | Description |
|--------|-------------|
| `k6_deploy_confirmation_ms_p50` / `_p95` | Confirmation latency percentiles |
| `k6_deploy_confirmed_total` | Deploys confirmed in a block |
| `k6_deploy_unconfirmed_total` | Deploys that timed out |

## Grafana: k6 Test Report Dashboard

Dashboard UID: `k6-test-report`, Title: "k6 Test Report".

**Template variable `$testid`**: queries `label_values(k6_http_reqs_total, testid)` with `refresh: 2` (on time range change), sorted descending so the latest run is first. Default time range: `now-24h`. Test run IDs are human-readable timestamps: `k6-hello-world-2026-03-30-11-10-05`.

**Annotations**: Region annotations from `annotateTestRun()` are shown as colored bands. Click a band to zoom the time range to that test run.

### Dashboard panels

| Panel | Type | Description |
|-------|------|-------------|
| Successful Deploys | stat (green) | `total - failed` requests for selected `$testid` |
| Failed Deploys | stat (red if >0) | Failed requests for selected `$testid` |
| Blocks Produced During Test Run | stat (blue) | `max - min` of `firefly_node_latest_block_number` over time range |
| CPU % by Node | timeseries | `node_cpu_seconds_total` idle mode, per node |
| Memory % by Node | timeseries | `node_memory_MemAvailable_bytes / MemTotal_bytes`, per node |

### Typical workflow

```
1. sudo make k6-run K6_SCRIPT=scenarios/confirmed-deploy K6_VUS=2 K6_DURATION=120s
2. make k6-logs  # watch progress
3. Open Grafana → Dashboards → "k6 Test Report"
4. Select testid from dropdown (e.g. k6-scenarios-confirmed-deploy-1711500000)
5. Panels show deploy rate, latency, confirmation time, CPU/RAM, block number
6. Annotation band marks the test window — click to zoom time range
```

## Adding a New Scenario

1. Create `k6/src/scenarios/my-scenario.js` (or `k6/src/my-scenario.js`)
2. Import shared lib:
   ```javascript
   import { deployAndCheck, getValidAfterBlockNumber, HELLO_WORLD_TERM } from "../lib/deploy.js";
   import { annotateTestRun } from "../lib/summary.js";
   ```
3. Export `options`, `setup()`, `default function`, and `handleSummary`:
   ```javascript
   export function handleSummary(data) {
     annotateTestRun(data, "my-scenario");
     return {};
   }
   ```
4. Rebuild: `sudo make k6-build`
5. Run: `sudo make k6-run K6_SCRIPT=scenarios/my-scenario`
